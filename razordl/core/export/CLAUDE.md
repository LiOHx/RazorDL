# Export layer — AST-based full-mode project export

Loaded when working in `razordl/core/export/`, or any preset's `_export.py` / `_export_full.py`.

## What it does

Generates self-contained projects (zero `razordl` dep) by AST-walking dependencies and rewriting `from razordl.xxx` imports to local imports.

## Per-preset export functions

Each preset exposes three AST-based functions (NEVER import preset code at module level — triggers heavy deps like tensordict, vllm):

- `export_workgroup(preset_pkg_dir)` → self-contained `workgroup.py` source
- `export_dataset(preset_pkg_dir)` → self-contained `dataset.py` source
- `export_full_project(preset_pkg_dir, project_dir, razordl_root)` → complete project

AST helpers in `ast_utils.py`:

- `extract_class(source, class_name)` — full class definition by line range
- `extract_function(source, function_name)` — top-level / helper function by line range
- `extract_standalone_classes(source)` — top-level classes only
- `extract_imports(source)` — all import statements (single- and multi-line)
- `replace_ident(source, old, new)` — word-boundary rename (does not match substrings)

## Engine export profiles

**Each engine owns a one-file `_export_profile.py`** that declares `engine_name` + `engine_files`. The actual export logic lives in `full_project.py` (shared across engines).

```python
# core/engine/single_model/_export_profile.py
from razordl.core.export.full_project import export_full_project_for_engine

def export_full_project(preset_pkg_dir, project_dir, razordl_root):
    return export_full_project_for_engine(
        preset_pkg_dir, project_dir, razordl_root,
        engine_name="single_model",
        engine_files=("main.py", "trainer.py", "workgroup.py", "config.py", "dataset.py"),
    )
```

Presets reference an engine profile via a one-line `_export_full.py`:

```python
# SFT / DFT / video_embedding
from razordl.core.engine.single_model._export_profile import export_full_project  # noqa: F401

# GRPO / OPD
from razordl.core.engine.on_policy_single_model._export_profile import export_full_project  # noqa: F401
```

## Path mapping (`full_project.py::_map_dest_path`)

- `core/engine/<engine_name>/` → `src/engine/`
- `core/engine/common/` → `src/engine/common/`
- `core/base/` → `src/base/`
- `ops/` → `src/ops/`
- Preset `config.py` → `src/config.py`
- Preset `workgroup.py` / `dataset.py` → regenerated by AST extraction (NOT copied)

## Dependency walker (`_find_razordl_imports`)

Records BOTH `node.module` AND `node.module + "." + alias.name` for every `from razordl.x import y`. Without this, submodule imports (e.g. `from razordl.core.base import checkpoint_info`) resolve to only the parent package, and the submodule is silently dropped from the export.

## Import rewriting

Lives in `full_project.py::rewrite_imports` (NOT in preset `_export_full.py` — that file is just a one-line re-export). Uses AST to locate import statement line ranges, then regex-replaces only those lines. Rewrites `from razordl.xxx` → local imports (e.g. `from engine.main import main`, `from base import checkpoint_info`). Engines do NOT cross-import each other — shared code is in `engine/common/` or `ops/`.

## Adding a new engine

1. Create `core/engine/<name>/_export_profile.py` with `engine_name=<name>` and the engine files list.
2. Presets using this engine point their `_export_full.py` at the new profile.
3. The shared rewriter already understands `engine/common/` and `ops/` — touch `full_project.py` only if you introduce a new top-level package.

## Preset-specific docstring handling

If a preset reuses SFT classes via AST extraction (e.g. `SFTModelGroup` in DFT/OPD), use string replacement to neutralize SFT-specific docstrings in generated code. **Never modify SFT source files directly** — they're authoritative SFT documentation.

## Hard rules

- **IMPORTANT — Generated code in `custom` / `full` mode must be self-contained.** Users should be able to read and modify every line without jumping to parent classes. AST extraction inlines what's needed.
- **IMPORTANT — Use `importlib.util.spec_from_file_location`** to load preset `_export.py` by file path. Never `import razordl.presets.<name>` at module level in CLI / init code.
- **IMPORTANT — Never hardcode import lists in `_export.py` headers.** Use `extract_imports()` to auto-derive imports from the source file. Hardcoded headers silently rot when the source file gains new dependencies (new ops packages, new third-party imports). The only manual work is handling renames (e.g. `WorkGroup` → `WorkGroup as _WorkGroup`) and filtering out cross-preset imports whose targets are inlined.
