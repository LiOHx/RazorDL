# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Three-tier architecture

RazorDL serves three user levels via `razordl init --preset <name> --mode <mode>`:

| Mode | Command | Generated output | Runtime dependency |
|------|---------|-----------------|--------------------|
| `simple` | `razordl train --config config.yaml --preset <name>` | config.yaml + run.sh + data/ | needs razordl installed |
| `custom` | `sh run.sh` ŌåÆ `python src/main.py` | config.yaml + run.sh + data/ + src/{dataset,workgroup,main}.py | needs razordl installed |
| `full` | `sh run.sh` ŌåÆ `python src/main.py` | complete independent project with all framework code inlined under src/ | zero dependency on razordl |

`run.sh` is always generated by `init.py` ŌĆö never read from templates.  The comments and commands must match the preset (e.g. `# Launch DFT training`, `--preset dft`).

## Framework directory structure

```
razordl/
  core/
    base/          # abstractions: Config, Trainer, WorkGroup, DataLoader, checkpoint_info
    engine/
      common/              # shared engine lifecycle: FSDPModelGroup, EngineWorkGroup,
                           #   EngineTrainer, main(), flat_config helper
      single_model/        # supervised single-model engine (SFT/DFT/video_embedding)
                           #   thin subclasses (SP split, chunked LM loss live in presets)
      on_policy_single_model/  # vLLM rollout + PPO/GRPO (reinforcement learning)
    export/        # AST-based full-mode project export (full_project, ast_utils)
  ops/             # razordl-independent utilities
    hardware/      # device detection (cuda.py for NVIDIA, device.py dispatches)
    distributed/   # all_gather_object etc.
    parallel/      # FSDP2, sequence parallel, activation offload
    model/         # HuggingFace tokenizer/model/attention builders, LoRA/PEFT
    loss/          # DistCrossEntropyLoss, distributed token normalization
    snapshot.py    # experiment code snapshot, hash, directory scanning
  presets/         # one directory per training recipe
    sft/           # standard supervised fine-tuning
    dft/           # confidence-weighted fine-tuning
    grpo/          # Group Relative Policy Optimization (math RL with vLLM)
    opd/           # PG On-Policy Distillation (vLLM rollout + teacher logŽĆ)
    video_embedding/  # video-text contrastive learning
    _template/     # (optional) copy target for new presets
  cli/             # argparse-based CLI (init, train, ckpt, diff ŌĆö no heavy deps at top level)
```

### Experiment directory structure

Each training run auto-creates a timestamped experiment directory::

    outputs/
      2026-05-26_14-30-00/              # auto-generated (outputs_dir/resume_mode in config)
        code/                           # frozen code snapshot (copy to resume elsewhere)
          config.yaml                   # output_dir hard-coded to absolute experiment path
          data/
          src/                          # custom/full mode
          run.sh
        pip_freeze.txt                  # full pip environment snapshot
        provenance.json                 # {code_hash, config_hash, git_commit, razordl_git_commit, ...}
        checkpoint_000500/
          checkpoint_info.json
          model.safetensors
          optimizer.pt
        checkpoint_001000/
        step_info.jsonl

- **Complete detection**: ``checkpoint_info.json`` at experiment root with ``kind: model_only`` means training finished (written by ``save_model_and_processor`` at end of training).
- **Resume logic** (``resume_mode: auto``): scan ``outputs/`` for latest experiment, check if complete, compare code hash + razordl version against ``provenance.json``.  Hash match + incomplete ŌåÆ auto-resume; hash mismatch ŌåÆ new experiment with warning.
- **Resume logic** (``resume_mode: manual``): ``resume_from`` required to resume; otherwise always new experiment.
- **Forking** (``init_from``): set ``init_from: <path-to-checkpoint>`` to fork a new experiment from an existing checkpoint ŌĆö model weights are loaded but optimizer state and step counter start fresh.  Always creates a new experiment directory.
- **Copy recovery**: copy ``code/`` to another location, ``config.yaml`` already points to original experiment dir via ``output_dir`` ŌåÆ ``razordl train`` resumes from the original checkpoints.

``razordl diff`` compares experiment code/config::

    razordl diff                              # current project vs latest experiment
    razordl diff outputs/2026-05-26_14-30-00  # current project vs specified experiment
    razordl diff outputs/<A> outputs/<B>       # experiment A vs experiment B

Output shows file changes (A/M/D), YAML config key diffs, and git provenance.
```

## Architecture hierarchy (base ŌåÆ engine/common ŌåÆ engine ŌåÆ preset)

The framework enforces a strict layered hierarchy.  **Engine-level concerns must be fixed at the shared engine layer, not worked around in presets or duplicated in engine variants.**

| Layer | Responsibility | Example |
|-------|---------------|---------|
| `core/base/` | Abstract interfaces, stable contracts, generic serialization | `BaseTrainer`, `BaseModelGroup`, `BaseConfig`, `checkpoint_info` |
| `core/engine/common/` | Shared training lifecycle for ALL engine variants | `FSDPModelGroup` (FSDP2/LoRA/resume/optimizer/save/load/offload), `EngineWorkGroup` (pre/post update wrapper, seeding), `EngineTrainer`, `main()`, `build_single_model_config_dict` |
| `core/engine/<name>/` | Engine variant: thin subclasses + paradigm-specific orchestration | `single_model.WorkGroup` (thin abstract wrapper), `on_policy_single_model.WorkGroup` (rollout/reward/advantage/loss template) |
| `presets/<name>/` | Task-specific logic only (data format, model family, loss/reward/rollout behavior) | `build_processor()`, `build_model()`, task-specific forward/loss/backward hooks |

**Rules:**
- Presets **inherit engine variant classes** (not `engine/common/*` or `base/*` directly).
- The shared lifecycle wrappers live in `engine/common/`:
  - `FSDPModelGroup.__init_subclass__` auto-wraps `build_model()` with `_post_build_model()` (adapter, resume, FSDP2/SP).
  - `EngineWorkGroup.__init_subclass__` auto-wraps `update_step()` (or `_run_update_step()` for on-policy) with `_pre_update_step()` (seeding, offload load) and `_post_update_step()` (optimizer step, grad clip, offload).
- Presets may implement task-specific forward/loss/backward behavior, but must **not** perform optimizer step, gradient clipping, parameter offloading, checkpoint save/load, or shared seeding.  Those are handled by the engine wrapper.
- LM-style presets (SFT/DFT) define their own `update_step` and loss computation (SP split, next-token CE, chunked loss) in the preset layer; the engine only provides the abstract `WorkGroup` + lifecycle wrappers.  On-policy presets should prefer the `rollout()` / `compute_reward()` / `compute_advantage()` / `compute_loss()` hooks; override `_run_update_step()` only when the task truly needs a different backward schedule (for example chunked GRPO loss).
- **Single point of change**: if FSDP2, LoRA, resume, save/load, grad-clip, offload, or per-step seeding needs to change, edit `engine/common/`.  Engine variants (`single_model/`, `on_policy_single_model/`) are thin and only carry paradigm differences.
- **Engine must NOT contain training logic.** Concrete forward passes, loss functions (CE, contrastive, etc.), SP split, chunked-loss monkey-patching, backward calls ŌĆö these all belong in **presets** (task-specific) or **ops** (stateless utilities shared across presets). Engine is purely abstract framework: lifecycle wrappers, distributed orchestration, training loop structure. If you write ``torch.nn.functional.cross_entropy`` (or any other concrete loss) in an engine file, it is in the wrong layer. This is the single most important architectural boundary in the project ŌĆö we moved ``LMWorkGroup`` out of engine to fix this exact violation.


### Dependency direction: tree, not graph

The framework is a tree ŌĆö **vertical inheritance is allowed, cross-branch imports are forbidden**::

    base  (core/base/)
    ŌööŌöĆŌöĆ engine/common  (shared engine lifecycle)
        Ōö£ŌöĆŌöĆ single_model  (core/engine/)          ŌåÉ engine branch
        Ōöé   Ōö£ŌöĆŌöĆ sft  (presets/)                  ŌåÉ preset leaf
        Ōöé   Ōö£ŌöĆŌöĆ dft        ŌåÉ inherits sft classes, OK (same branch)
        Ōöé   ŌööŌöĆŌöĆ video_embedding
        ŌööŌöĆŌöĆ on_policy_single_model  (core/engine/)
            Ōö£ŌöĆŌöĆ grpo  (presets/)
            ŌööŌöĆŌöĆ opd   (presets/)

**Vertical inheritance ŌĆö allowed.**
Base ŌåÆ engine/common ŌåÆ engine variant ŌåÆ preset is the canonical chain.
Within presets, DFT inheriting from SFT is fine because both live under the
same engine.

Cross-branch **imports ŌĆö forbidden.**
``on_policy_single_model`` must not ``import`` anything from ``single_model``
(and vice versa).  Presets under different engines must not cross-import either.
Doing so drags unrelated files into full-mode export and causes hard-to-debug
file collision / missing-code bugs.

**Shared code lives in `engine/common/` or `ops/`.**
- Lifecycle/engine state (FSDP2 ModelGroup, EngineWorkGroup wrapper, EngineTrainer, flat config helper) ŌåÆ `core/engine/common/`.
- Stateless utilities (LoRA adapter loading, HF model/tokenizer builders, distributed loss helpers) ŌåÆ `ops/`.
- Never let one engine variant import from another to share code ŌĆö promote the shared piece to `engine/common/` or `ops/` instead.

## Preset conventions (convention over configuration)

Every preset directory under `razordl/presets/<name>/` must contain:

```
<name>/
  __init__.py          # exports canonical classes + CLI CamelCase aliases
  config.py            # canonical config class (e.g. SFTConfig, GRPOConfig)
  workgroup.py         # canonical WorkGroup + preset-specific model/loss/reward logic
  _export.py           # AST-based code generation (export_workgroup, export_dataset)
  _export_full.py      # thin re-export of SFT's export_full_project
  default_config.yaml  # complete template with ALL params, [ÕĖĖńö©]/[õĖŹÕĖĖńö©] sections
  requirements.txt     # auto-synced from pyproject.toml, do not edit manually
```

**Naming rules** (enforced by CLI auto-discovery):
- The CLI converts the preset name with `CamelCase = "".join(part.capitalize() for part in preset.split("_"))`.
- Each preset module must expose CLI aliases named `{CamelCase}Config`, `{CamelCase}WorkGroup`, `{CamelCase}Dataset`, `{CamelCase}Collator`.
- Canonical class names may preserve acronyms for readability (e.g. `SFTConfig`, `GRPOWorkGroup`), but `__init__.py` must provide aliases (`SftConfig`, `GrpoWorkGroup`) for the CLI.
- Preset module path is always `razordl.presets.{preset}` (e.g. `razordl.presets.dft`).

**CLI auto-discovery**: `cli/__init__.py` scans `razordl/presets/` at startup.  Directories starting with `_` are excluded.  `--preset` choices and error messages are generated dynamically.  No hardcoded preset names anywhere in CLI code.

**When adding a new preset**, copy `presets/_template/` and edit files marked `[µö╣]`.  The CLI discovers it automatically.  No changes to `cli/__init__.py`, `cli/init.py`, or `cli/train.py` are needed.  See `presets/_template/README.md` for step-by-step instructions.

### Flat config (YAML) ŌåÆ nested config

Every preset's `from_flat_dict(d: dict)` MUST delegate to `razordl.core.engine.common.flat_config.build_single_model_config_dict()` for the shared mapping (model/optimizer/trainer/LoRA/offload/ray_kwargs/`log_steps * grad_accum`/`compute_checksums`/etc.).  Presets only declare:
- their own `data_config` dict (task-specific keys like GRPO's `group_size`/`kl_coef`, video_embedding's `nframes`)
- their default `model_default`, `lr_default`, `grad_accum_default`, `log_steps_default`, `task_type_default`

Never copy the trainer/model/optimizer mapping into a preset.  Adding a new public flat key means editing `engine/common/flat_config.py` once.

### Checkpoint metadata

Every saved checkpoint contains a `checkpoint_info.json` (rich metadata) instead of the legacy single-integer `complete` marker.  Schema lives in `razordl/core/base/checkpoint_info.py`.  Includes: `completed_step`, `timestamp`, `elapsed_seconds`, `topology` (world_size / sp_size), `seed`, `provenance` (`framework_version`, `git_commit`, `config_hash`), `metrics` (latest aggregated step_info), `files` (size + optional SHA256), `resumed_from`, `kind` (`checkpoint` or `model_only`).  `framework_version` is `"razordl"`'s version when running in this repo; in a full-mode exported project the brand constant is rewritten to `None`, so generated projects log `framework_version: "unknown"` and never advertise the upstream package name.

- `is_complete()` accepts BOTH `checkpoint_info.json` AND legacy `complete` for backwards compatibility.
- On resume, `_check_topology_compat()` warns (does not raise) if `world_size`/`sp_size` differ from what the checkpoint was saved with.
- `trainer_config.compute_checksums: bool = False` ŌĆö opt-in SHA256 over model files at save time.  File sizes are always recorded.
- Inspect any checkpoint with `razordl ckpt info <dir>`.

### Distributed metrics (Reducible)

`step_info` returned by `update_step` is gathered across ranks/micro-steps and collapsed by `_summarize_step_info` (`core/base/trainer.py`).  Plain scalars (loss, grad_norm, clip_fraction, ...) are reduced by mean ŌĆö fine.  But **distribution-shaped fields** (where min/max/std are meaningful) MUST NOT be pre-aggregated to scalars on a rank: "mean of per-rank max" is meaningless and produces values outside the true sample set.

The fix is a `Reducible` ABC in `razordl/core/base/metrics.py`.  Subclasses carry sufficient statistics and own their own cross-rank merge:

- `DistStats.from_tensor(t)` ŌĆö emits `{sum, sum_sq, n, min, max}` and unfolds to `{mean, std, min, max, n}` after merge.  Use this whenever you would have computed `{mean, std, min, max}` locally.  `std` is population std.
- Adding a new distribution kind (histogram, ratio, weighted mean) means subclassing `Reducible`; the aggregator is not touched.

`_summarize_step_info` handles a single `Reducible` leaf and a list-of-same-type `Reducible` (the post-`all_gather_object` shape).  Gathering happens exactly once, in `BaseTrainer.run_training_loop` ŌĆö `EngineTrainer.update_step` deliberately does NOT call `all_gather_object` (a second gather would inflate `DistStats.n` by `world_size`).

## Engine export profiles

Each engine under `core/engine/` owns a one-file `_export_profile.py` whose only job is to declare `engine_name` and `engine_files` (the engine-specific .py files to seed the dependency walker with).  The actual export logic ŌĆö AST dependency collection, import rewriting, path mapping, dataset/workgroup AST extraction, generated `main.py` ŌĆö lives in `razordl/core/export/full_project.py` (shared by all engines).

```python
# razordl/core/engine/single_model/_export_profile.py
from razordl.core.export.full_project import export_full_project_for_engine

def export_full_project(preset_pkg_dir, project_dir, razordl_root):
    return export_full_project_for_engine(
        preset_pkg_dir, project_dir, razordl_root,
        engine_name="single_model",
        engine_files=("main.py", "trainer.py", "workgroup.py", "config.py", "dataset.py"),
    )
```

Presets reference an engine profile via a one-line `_export_full.py`:
```python
# SFT / DFT / video_embedding
from razordl.core.engine.single_model._export_profile import export_full_project  # noqa: F401

# GRPO (on-policy RL)
from razordl.core.engine.on_policy_single_model._export_profile import export_full_project  # noqa: F401
```

**Path mapping** (in `core/export/full_project.py::_map_dest_path`): `core/engine/<engine_name>/` ŌåÆ `src/engine/`, `core/engine/common/` ŌåÆ `src/engine/common/`, `core/base/` ŌåÆ `src/base/`, `ops/` ŌåÆ `src/ops/`.  Presets' `config.py` ŌåÆ `src/config.py`; preset `workgroup.py`/`dataset.py` are regenerated by AST extraction (not copied).

**Dependency walker** (`_find_razordl_imports`): records BOTH `node.module` AND `node.module + "." + alias.name` for every `from razordl.x import y` so submodule imports (e.g. `from razordl.core.base import checkpoint_info`) get resolved as `razordl.core.base.checkpoint_info`.  Without this, importing a submodule via the parent-package form would silently drop the submodule from the export.

When adding a new engine (e.g. `multi_model`):
1. Create `core/engine/<name>/_export_profile.py` with `engine_name=<name>` and the list of engine files.
2. Presets using this engine point their `_export_full.py` at the new profile.
3. The shared rewriter already understands `engine/common/` and `ops/` ŌĆö no need to touch `core/export/full_project.py` unless you introduce a new top-level package.

## Dependency management

- **Single source of truth**: `pyproject.toml` ŌåÆ `[project.dependencies]` (shared core) + `[project.optional-dependencies]` (per-preset extras)
- **Auto-sync**: `scripts/sync_requirements.py` generates `presets/*/requirements.txt` by merging `[project.dependencies]` with each preset's optional extra
- **Root `requirements.txt`**: deleted ŌĆö `pip install -e .` uses pyproject.toml directly
- **pre-commit hook**: `.pre-commit-config.yaml` runs the sync script when `pyproject.toml` changes
- **Preset extras convention**: match the preset name exactly (e.g. `[project.optional-dependencies] sft = [...]`)
- **Preset requirements.txt**: auto-generated file, top comment says "Auto-generated from pyproject.toml ŌĆö Do not edit manually"

To add a preset-specific dependency:
1. Add it to `pyproject.toml` under `[project.optional-dependencies].<preset>`
2. Commit ŌĆö the pre-commit hook syncs `presets/<preset>/requirements.txt`

## Code generation (_export.py pattern)

Two export functions per preset, both using **AST extraction** (never importing preset code at module level):

- `export_workgroup(preset_pkg_dir)` ŌåÆ self-contained workgroup.py source
- `export_dataset(preset_pkg_dir)` ŌåÆ self-contained dataset.py source
- `export_full_project(preset_pkg_dir, project_dir, razordl_root)` ŌåÆ complete independent project

AST helpers live in `razordl/core/export/ast_utils.py` and are reused by preset `_export.py` files:
- `extract_class(source, class_name)` ŌĆö extract full class definition by line range
- `extract_function(source, function_name)` ŌĆö extract a top-level/helper function by line range
- `extract_standalone_classes(source)` ŌĆö extract top-level classes only
- `replace_ident(source, old, new)` ŌĆö word-boundary rename (will not match substrings)

**Import rewriting** lives in `razordl/core/export/full_project.py::rewrite_imports` (NOT in preset `_export_full.py` ŌĆö that file is just a one-line re-export).  Uses AST to locate import statement line ranges, then regex-replaces only those lines.  Rewrites `from razordl.xxx` ŌåÆ local imports (e.g. `from engine.main import main`, `from base import checkpoint_info`).  Engines do NOT cross-import each other anymore ŌĆö shared code is promoted to `engine/common/` or `ops/`.

**Preset-specific docstring handling** (`_export.py` in non-sft presets): If the preset reuses SFT classes via AST extraction (e.g. `SFTModelGroup`), use string replacement to neutralize SFT-specific docstrings in generated code.  Never modify SFT source files directly.

## Hardware detection

`ops/hardware/` ŌĆö extendable device detection:
- `cuda.py` ŌĆö CUDA-specific: version, driver compatibility, recommended PyTorch index
- `device.py` ŌĆö dispatches to backends, `get_available_device()` returns "cuda"/"mps"/"cpu"
- `check_device_compatibility()` ŌĆö called at training startup; raises RuntimeError with install guidance
- Uses `importlib.import_module` for lazy loading to avoid circular imports during package init
- Adding a new backend (ROCm, XPU) only requires a new file + one line in `device.py`

## Key invariants

- **No hardcoded preset names in CLI code** ŌĆö use `os.listdir("razordl/presets")` and convention-based derivation
- **init.py's main.py generation** must derive `config_class` from the same CamelCase convention as CLI train (`"".join(part.capitalize() for part in preset.split("_")) + "Config"`) and `preset_module = f"razordl.presets.{preset}"` ŌĆö no if/else per preset
- **train.py** uses `importlib.import_module(f"razordl.presets.{preset}")` + `getattr(mod, f"{CamelCase}Config")` / `WorkGroup` / `Dataset` / `Collator` for dynamic dispatch
- **_export_full.py** in presets is always a one-line re-export pointing at the engine's export profile: `from razordl.core.engine.<engine_name>._export_profile import export_full_project`
- **Never import preset packages at module level** in CLI or init code (triggers heavy deps like tensordict).  Use `importlib.util.spec_from_file_location` to load _export.py by file path
- **Generated code in custom/full mode** must be self-contained ŌĆö user should be able to read and modify every line without jumping to parent classes
- **docstrings in SFT source files** describe SFT.  When other presets reuse SFT classes via AST export, they must strip/replace SFT-specific docstring text in the generated output
- **requirements.txt is auto-generated** ŌĆö never commit manual edits to preset requirements.txt files
- **PyTorch minimum version**: >=2.6.0 (FSDP2 API stabilized), enforced in pyproject.toml
- **Architecture hierarchy is absolute**: `base ŌåÆ engine/common ŌåÆ engine variant ŌåÆ preset`.  Presets inherit engine variant classes, never `engine/common/*` or `base/*` directly.  Engine bugs are fixed in `engine/common/` (single source of truth for FSDP/LoRA/resume/optimizer/grad-clip/offload/seeding); engine variants are thin paradigm-specific wrappers.
- **Shared engine logic has ONE home: `engine/common/`.**  Never duplicate FSDP2/LoRA/resume/optimizer/save/load code across engine variants ŌĆö if a variant needs to vary, expose a hook in `engine/common/`.  Never let one engine variant import from another (e.g. `on_policy_single_model` importing from `single_model`) ŌĆö promote shared code to `engine/common/` or `ops/`.
- **Engine is pure framework, never training logic.** Engine files under `core/engine/` contain only abstract lifecycle infrastructure ŌĆö wrappers, training loop structure, distributed orchestration, optimizer step, checkpointing. They must **never** contain concrete forward passes, loss functions (cross-entropy, contrastive, etc.), SP split logic, chunked-loss implementations, or backward calls. Those go in **presets** (task-specific training recipes) or **ops** (stateless utilities). The `LMWorkGroup` that previously lived in `engine/single_model/lm_workgroup.py` was a violation of this rule ŌĆö it was deleted and all its logic moved into `presets/sft/workgroup.py::SFTWorkGroup`. Before adding any code to an engine file, ask: "Is this a specific training behavior, or is it generic infrastructure?" If the answer is "specific training behavior," it belongs in a preset.
- **Shared YAML key mapping has ONE home: `engine/common/flat_config.py::build_single_model_config_dict`.**  Presets call it with their data-config + defaults; never copy the trainer/optimizer/LoRA mapping into a preset.
- **Checkpoint marker is `checkpoint_info.json`** (rich JSON metadata), not the legacy `complete` file.  All checkpoint detection paths (`is_complete`, `get_resume_checkpoint_dir`, `get_resume_state`) accept both for backwards compatibility, but new saves only write `checkpoint_info.json`.
- **Auto post-init wrapping uses a depth counter** (`AutoSetModelGroupNameWorkGroup` in `core/base/workgroup.py`): every subclass `__init__` is wrapped, but `__post_init__` fires only when the outermost wrapper returns (depth ŌåÆ 0).  This is critical because subclasses like `SFTWorkGroup` create `self.model_group` AFTER `super().__init__()` returns ŌĆö a naive one-shot guard would fire `__post_init__` before the attribute exists and `auto_set_model_group_name` would silently walk an empty set.
- **Distribution metrics use `Reducible` leaves.**  Never pre-aggregate min/max/std/sum into a scalar on a rank and ship that ŌĆö `_summarize_step_info` will then average those scalars across ranks and produce values outside the true sample set (e.g. GRPO `reward.max=1.0` when rewards are `{0.0, 0.5, 1.5}`).  Use `DistStats.from_tensor(t)` (or another `Reducible` subclass in `core/base/metrics.py`) so the aggregator can pool exactly.  Engine-layer gather happens only in `BaseTrainer.run_training_loop`; do not re-add `all_gather_object` in `EngineTrainer.update_step` or `DistStats.n` will be inflated by `world_size`.
- **Frozen ModelGroups are never persisted.**  `FSDPModelGroup.save_model_and_processor()` and `save_checkpoint()` early-return when `is_trainable=False`.  Reference / teacher models therefore do NOT appear in `checkpoint_*/` subdirs ŌĆö only `policy_model_group/` does.  When designing new on-policy presets, reuse the engine's `reference_model_group` slot for any frozen counterpart (teacher, critic, etc.) rather than introducing a parallel `model_group_config`.
- **OPD requires `data_config.teacher_model`.**  Student and teacher MUST share the same tokenizer (vocab + special-token ids).  `OPDWorkGroup.__init__` enforces this with `_validate_tokenizer_compat()`.  The teacher path lives on `data_config` (not a second `model_group_config`); `OPDWorkGroup.__init__` deep-copies the policy config and overrides `model_path`/`processor_path`/`is_trainable=False` for the teacher.
- **Experiment management: `outputs_dir` + `resume_mode`.**  Users configure ``outputs_dir`` (parent) and ``resume_mode: auto|manual`` ŌĆö ``output_dir`` is set by the framework at experiment creation time and must NOT appear in user configs.  ``resume_mode: auto`` scans ``outputs/`` for the latest experiment, compares code hash + razordl version against ``provenance.json``, and auto-resumes when unchanged and training is incomplete; otherwise creates a new experiment.  ``resume_mode: manual`` always creates a new experiment unless ``resume_from`` points to an existing experiment.  ``init_from: <checkpoint-path>`` forks a new experiment from an existing checkpoint ŌĆö model weights load from the checkpoint, but optimizer and step start fresh.
- **Deterministic training: ``set_seed``.**  ``BaseTrainer.set_seed(seed)`` is the single entry point for seeding Python/numpy/torch/CUDA.  It is called before model initialization (for deterministic LoRA/weight init) and before every training step via ``_pre_update_step``.  Never add ad-hoc ``random.seed`` / ``torch.manual_seed`` calls elsewhere ŌĆö always import and call ``set_seed``.
- **Every preset's `build_model()` MUST go through `razordl.ops.model.huggingface.enforce_model_profile(model_path)`.**  This looks up the checkpoint's `cfg.model_type` in `razordl/ops/model/profiles/` and raises `UnsupportedModelError` if no profile is registered, preventing deep-stack crashes from unsupported models.  The function returns the (possibly-adjusted) `PreTrainedConfig` to pass via `from_pretrained(config=...)`.  LM-style presets get this automatically through `build_causal_lm`; multimodal presets (video_embedding) call it explicitly.  To add support for a new model family: create `razordl/ops/model/profiles/<model_type>.py` with a `@register` decorated `ModelProfile` subclass, then add a corresponding absolute-import line in `profiles/__init__.py` (relative imports break the full-mode export's AST walker).  Profiles are stateless, idempotent, and only handle HF-load-time concerns (config preparation, hard precondition checks) ŌĆö never training logic.

## Development environment

The project uses a **uv-managed** virtual environment at `.venv/`.

- **Activate**: `source .venv/bin/activate`
- **Install packages**: `uv pip install <pkg>` (NOT `pip install`)
- **Python**: `.venv/bin/python`

### Adding dependencies

Always use `uv pip` inside the venv. Examples:

```bash
source .venv/bin/activate
uv pip install torchvision --index-url https://download.pytorch.org/whl/cu126
MAX_JOBS=4 uv pip install flash-attn --no-build-isolation
```

### Torch / flash-attn compatibility

`flash-attn` must be compiled against the installed PyTorch version. If you see `undefined symbol: _ZN3c104cuda...`, the ABI is mismatched. Fix:

```bash
uv pip uninstall flash-attn
uv pip install "torch==2.6.0+cu126" --index-url https://download.pytorch.org/whl/cu126
MAX_JOBS=4 uv pip install flash-attn --no-build-isolation
```

### Ray dashboard

`pyproject.toml` declares dashboard deps (aiohttp-cors, grpcio, opencensus, opentelemetry-*, prometheus-client) alongside `ray[train]`.  Without them, Ray starts dashboard in minimal mode, State API is unavailable, and `PlacementGroupCleaner` spams ``Failed to query Ray Train Controller actor state`` every second.

If all deps are installed, do **not** set ``include_dashboard: false`` ŌĆö that also disables State API and triggers the same spam.
